# 代理协调器 (Agent Orchestrator)

通过将复杂任务分解为子任务、生成自主子代理并整合其工作成果，来高效编排复杂任务。

## 核心工作流程

### 第 1 阶段：任务分解 (Task Decomposition)

分析宏观任务并将其分解为相互独立、可并行的子任务。

**分解原则：**
- 每个子任务应该是可以独立完成的。
- 最小化代理之间的依赖关系。
- 倾向于分配职责宽泛、自主性强的任务，而非狭窄、相互依赖的任务。
- 为每个子任务设定明确的成功标准。

### 第 2 阶段：代理生成 (Agent Generation)

为每个子任务创建一个子代理工作区。这包括：

**动态生成 `SKILL.md`：**
- 定义代理的具体角色和目标。
- 列出所需的工具和能力。
- 规定输入/输出规范。
- 设定成功标准。
- 制定通信协议。

### 第 3 阶段：代理派遣 (Agent Dispatch)

通过以下方式初始化并启动每个代理：
1. 向任务指令文件写入具体指令。
2. 将所需的输入文件复制到代理的工作目录。
3. 设置必要的环境变量。
4. 使用任务工具孵化并运行代理。

### 第 4 阶段：监控与检查点 (Monitoring & Checkpoints)

对于完全自主的代理，仅需最少的监控：
定期检查每个代理的状态文件。代理在完成任务后会更新此文件。

### 第 5 阶段：整合 (Integration)

在所有代理任务完成后：
1. 从每个代理的输出目录收集成果。
2. 根据成功标准验证交付成果。
3. 根据需要合并或集成输出内容。
4. 解决可能出现的冲突（如多个代理修改了同一部分）。
5. 生成所有已完成工作的总结报告。

### 第 6 阶段：解散与总结 (Dissolution & Summary)

整合完成后：
1. 存档代理工作区（可选）。
2. 清理临时文件。
3. 生成最终总结：
   - 每个代理完成了什么工作。
   - 遇到的任何问题及解决情况。
   - 最终交付物的存储位置。
   - 时间与资源消耗指标。

## 基于文件的通信协议

- **指令文件**：代理只读，由协调器（Orchestrator）编写。
- **输出文件**：代理只写，由协调器读取。
- **状态文件**：代理实时更新，用于报告进度（例如：`todo` -> `in_progress` -> `done` | `failed`）。

## 子代理模板示例

预置的常见代理类型模板包括：
- **研究代理 (Research Agent)** - 负责网络搜索、数据收集。
- **代码代理 (Code Agent)** - 负责实现代码与测试。
- **分析代理 (Analysis Agent)** - 负责数据处理与模式发现。
- **撰写代理 (Writer Agent)** - 负责内容创作与文档编写。
- **审核代理 (Review Agent)** - 负责质量保证与编辑审核。
- **集成代理 (Integration Agent)** - 负责合并输出与解决冲突。

## 最佳实践

1. **从小规模开始** - 先从 2-3 个代理开始尝试，随着模式成熟再逐步扩展。
2. **界限清晰** - 确保每个代理都有特定的交付物所有权。
3. **显式移交** - 使用结构化文件进行代理间的通信和任务移交。
4. **优雅失败** - 代理应主动报告失败，由协调器负责恢复或重试逻辑。
5. **完整记录** - 通过状态文件全程跟踪调试进度。
