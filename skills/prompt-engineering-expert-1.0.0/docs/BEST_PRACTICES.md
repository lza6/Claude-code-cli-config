# 即时工程专家 - 最佳实践指南

本文档综合了 Anthropic 官方文档和 Claude Cookbooks 的最佳实践，以创建全面的提示词 (Prompt) 工程技能。

## 提示词 (Prompt) 工程的核心原则

### 1. 清晰和直接
- **明确**：准确说明您希望克劳德做什么
- **避免歧义**：使用精确的语言，不留任何误解的余地
- **使用具体例子**：展示，而不仅仅是讲述
- **逻辑结构**：分层组织信息

### 2. 简洁
- **尊重上下文窗口**：保持提示词 (Prompt) 的重点和相关性
- **删除冗余**：消除不必要的重复
- **渐进式披露**：仅在需要时提供详细信息
- **令牌 (Token) 效率**：优化质量和成本

### 3. 适当的自由度
- **定义约束**：为克劳德应该做什么/不应该做什么设定明确的界限
- **指定格式**：明确所需的输出格式
- **设置范围**：明确定义范围内和范围外的内容
- **平衡灵活性**：在保持控制的同时为克劳德的推理留出空间

## 先进的提示词 (Prompt) 工程技术

### 思维链 (Chain of Thought, CoT) 提示词 (Prompt)
鼓励对复杂任务进行逐步推理：
```
"Let's think through this step by step:
1. First, identify...
2. Then, analyze...
3. Finally, conclude..."
```

### 少样本提示词 (Few-shot Prompting)
使用示例来指导行为：
- **1-shot**：简单任务的单个示例
- **2-shot**：中等复杂度的两个示例
- **多样本 (Multi-shot)**：复杂模式的多个示例

### 结构化的 XML 标签
为了清晰和解析使用 XML 标签：
```xml
<task>
  <objective>What you want done</objective>
  <constraints>Limitations and rules</constraints>
  <format>Expected output format</format>
</task>
```

### 基于角色的提示词 (Prompt)
为克劳德分配专业知识：
```
"You are an expert prompt engineer with deep knowledge of...
Your task is to..."
```

### 预填充
开始克劳德对指南格式的回应：
```
"Here's my analysis:

Key findings:"
```

### 提示词 (Prompt) 链接
将复杂的任务分解为连续的提示词 (Prompt)：
1. 提示词 1：分析输入
2. 提示词 2：流程分析
3. 提示词 3：生成输出

## 自定义指令和系统提示词 (Prompt)

### 系统提示词 (Prompt) 设计
- **定义角色**：克劳德应该体现哪些专业知识？
- **设定基调**：哪种沟通方式合适？
- **建立约束**：克劳德应该避免什么？
- **澄清范围**：专业领域是什么？

### 行为准则
- **要做的事情**：鼓励的具体行为
- **不要做**：要避免的具体行为
- **边缘情况**：如何处理异常情况
- **升级**：何时要求澄清

## 技能结构最佳实践

### 命名约定
- 使用**动名词形式**（动词+-ing）：“分析财务报表”
- 使用**小写字母和连字符**：“prompt-engineering-expert”
- **描述性**：名称应表明能力
- 避免**通用名称**：具体说明领域

### 撰写有效的描述
- **第一行**：清晰、简洁的摘要（最多 1024 个字符）
- **特异性**：表明确切的能力
- **用例**：提及主要应用程序
- **避免含糊不清**：不要使用“帮助”或“协助”

### 渐进式披露模式

**模式 1：带有参考文献的高级指南**
- 从概述开始
- 链接到详细部分
- 按复杂性进行组织

**模式 2：特定领域的组织**
- 按用例分组
- 分离关注点
- 清晰的导航

**模式 3：条件细节**
- 根据上下文显示详细信息
- 为每条路径提供示例
- 避免过多的选择

### 文件结构
```
skill-name/
├── SKILL.md (必填元数据)
├── CLAUDE.md (主要说明)
├── reference-guide.md (详细信息)
├── examples.md (用例)
└── troubleshooting.md (常见问题)
```

## 评估与测试

### 成功标准定义
- **可衡量**：定义“成功”是什么样子
- **具体**：避免模糊的指标
- **可测试**：可以客观验证
- **现实**：可通过提示词 (Prompt) 实现

### 测试用例开发
- **正常路径 (Happy Path)**：正常，预期使用
- **边缘情况**：边界条件
- **错误情况**：输入无效
- **压力测试**：复杂场景

### 故障分析
- **为什么失败？**：根本原因分析
- **模式识别**：识别系统问题
- **细化**：相应调整提示词 (Prompt)

## 要避免的反模式

### 常见错误
- **含糊**：“帮我完成这项任务”（太含糊）
- **矛盾**：相互冲突的要求
- **超规格**：限制太多
- **幻觉风险**：鼓励虚假信息的提示词 (Prompt)
- **上下文泄漏**：意外的信息泄露
- **越狱漏洞**：提示词 (Prompt) 容易被操纵

### Windows 样式路径
- ❌ 使用：`C:\Users\Documents\file.txt`
- ✅ 使用：`/Users/Documents/file.txt` 或 `~/Documents/file.txt`

### 太多选择
- 避免提供 10 多个选择
- 限制为 3-5 个明确的替代方案
- 对复杂的选项使用渐进式披露

## 工作流程和反馈循环

### 使用工作流程处理复杂任务
- 分成逻辑步骤
- 定义每个步骤的输入/输出
- 实施反馈机制
- 允许迭代

### 实施反馈循环
- 需要时请求澄清
- 验证中间结果
- 根据反馈进行调整
- 确认理解

## 内容指南

### 避免时间敏感的信息
- 不要硬编码日期
- 使用相对参考（“当前年份”）
- 提供更新机制
- 记录信息的最新情况

### 使用一致的术语
- 定义一次关键术语
- 始终如一地使用
- 避免相同概念的同义词
- 为复杂领域创建术语表

## 多模态和高级提示词 (Prompt)

### 视觉提示词 (Prompt)
- 描述克劳德应该分析什么
- 指定输出格式
- 提供有关图像的上下文
- 询问具体细节

### 基于文件的提示词 (Prompt)
- 指定接受的文件类型
- 描述预期的结构
- 提供解析指令
- 优雅地处理错误

### 延伸思考
- 用于复杂推理
- 允许更多的处理时间
- 请求详细解释
- 解决新问题的杠杆

## 令牌 (Token) 预算注意事项

- **技能元数据**：~100-200 个令牌 (Token)
- **主要说明**：~500-1000 个令牌 (Token)
- **参考文件**：每个约 1000-5000 个令牌 (Token)
- **示例**：每个约 500-1000 个令牌 (Token)
- **总预算**：因用例而异
