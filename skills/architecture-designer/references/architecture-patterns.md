# 架构模式

## 模式对比

| 模式 | 最适合 | 团队规模 | 权衡 |
|---------|----------|-----------|------------|
| **单体架构 (Monolith)** | 简单领域、小团队 | 1-10 人 | 部署简单；难以对部分进行扩展 |
| **模块化单体 (Modular Monolith)** | 复杂度增加 | 5-20 人 | 模块边界清晰；仍为单一部署 |
| **微服务 (Microservices)** | 复杂领域、大组织 | 20 人以上 | 独立扩展；运维复杂度高 |
| **无服务器 (Serverless)** | 可变负载、事件驱动 | 任何规模 | 自动扩展；冷启动、供应商锁定 |
| **事件驱动 (Event-Driven)** | 异步处理 | 10 人以上 | 松耦合；调试复杂度高 |

## 单体架构 (Monolith)

```
┌─────────────────────────────────────┐
│               应用程序              │
│  ┌─────┐  ┌─────┐  ┌─────┐          │
│  │用户 │  │订单 │  │产品 │          │
│  └─────┘  └─────┘  └─────┘          │
│  └──────────┬──────────────┘        │
│             数据库                  │
└─────────────────────────────────────┘
```

**何时使用**：
- 启动新项目时
- 小团队（< 10 名开发人员）
- 领域逻辑简单
- 需要快速迭代

**优点**：部署简单，易于调试，无网络延迟
**缺点**：难以独立扩展，技术受限，部署风险较高

## 微服务 (Microservices)

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务 │  │ 订单服务 │  │ 产品服务 │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
┌────▼────┐  ┌────▼────┐  ┌────▼────┐
│ 用户库   │  │ 订单库   │  │ 产品库   │
└─────────┘  └─────────┘  └─────────┘
```

**何时使用**：
- 大团队（20 名以上开发人员）
- 具有清晰边界的复杂领域
- 不同服务有不同的扩展需求
- 多语言技术栈需求

**优点**：独立扩展，团队自主，故障隔离
**缺点**：分布式系统复杂度，最终一致性，运维开销

## 事件驱动 (Event-Driven)

```
┌──────────┐     ┌─────────────┐     ┌──────────┐
│ 生产者   │────▶│ 消息总线    │────▶│ 消费者   │
└──────────┘     │ (如 Kafka)  │     └──────────┘
                 └─────────────┘
                       │
                       ▼
                 ┌──────────┐
                 │ 消费者   │
                 └──────────┘
```

**何时使用**：
- 需要异步处理
- 服务之间松耦合
- 事件溯源需求
- 高吞吐量消息传递

**优点**：解耦服务，可扩展，审计跟踪
**缺点**：最终一致性，调试复杂度，消息顺序保证

## CQRS (命令查询职责分离)

```
┌─────────┐         ┌─────────────┐
│  命令   │────────▶│  写模型     │──┐
└─────────┘         └─────────────┘  │
                                     ▼
                              ┌──────────┐
                              │  事件    │
                              └──────────┘
                                     │
┌─────────┐         ┌─────────────┐  │
│  查询   │◀────────│  读模型     │◀─┘
└─────────┘         └─────────────┘
```

**何时使用**：
- 读写比例严重倾斜
- 复杂的读取查询
- 事件溯源架构
- 不同的优化需求

## 快速参考

| 需求 | 推荐模式 |
|-------------|---------------------|
| 简单的 CRUD 应用 | 单体架构 |
| 发展中的创业公司 | 模块化单体 |
| 企业级规模 | 微服务 |
| 可变负载 | 无服务器 |
| 异步处理 | 事件驱动 |
| 读密集型 | CQRS |
