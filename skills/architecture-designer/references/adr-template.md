# ADR 模板

## ADR 格式

```markdown
# ADR-{number}: {标题}

## 状态
[Proposed（建议） | Accepted（已接受） | Deprecated（已废弃） | Superseded by ADR-XXX（被 ADR-XXX 取代）]

## 背景
[描述当时的情况和影响因素。问题是什么？
存在哪些约束？我们试图实现什么目标？]

## 决策
[明确陈述决策。我们打算怎么做？]

## 后果

### 正面影响
- [收益 1]
- [收益 2]

### 负面影响
- [弊端 1]
- [弊端 2]

### 中性影响
- [既非好事也非坏事的副作用]

## 已考虑的备选方案
[评估了哪些其他选项，以及为什么拒绝它们？]

## 参考资料
- [相关文档链接]
- [讨论/RFC 链接]
```

## 示例：数据库选择

```markdown
# ADR-001: 使用 PostgreSQL 作为主数据库

## 状态
Accepted（已接受）

## 背景
我们需要为我们的电子商务平台提供一个关系型数据库，要求：
- 处理具有强一致性的复杂事务
- 支持 JSON 以实现灵活的产品属性
- 可扩展至数百万个产品和订单
- 与我们现有的 Python/Node 技术栈配合良好

团队拥有 PostgreSQL 和 MySQL 的经验。
预算允许使用托管数据库服务。

## 决策
使用 PostgreSQL 作为主数据库，托管在 AWS RDS 上。

## 后果

### 正面影响
- 财务交易符合 ACID 特性
- 功能集丰富（JSON、全文搜索、CTE）
- 强大的社区和工具支持
- 配合适当索引具有出色的性能
- 免费且开源

### 负面影响
- 垂直扩展有限（通过读取副本解决）
- 优化需要 DBA 专业知识
- 高可用性产生的 AWS RDS 成本

### 中性影响
- 团队需要学习 PostgreSQL 特有的功能
- 需要从当前的 SQLite 开发数据库进行迁移

## 已考虑的备选方案

**MySQL**
- 拒绝原因：在 JSON 操作方面功能较少
- 考虑因素：成本相似，团队熟悉

**MongoDB**
- 拒绝原因：订单/库存需要关系型数据模型
- 考虑因素：产品目录灵活性很高

**CockroachDB**
- 拒绝原因：成本较高，团队不熟悉
- 考虑因素：更好的水平扩展能力

## 参考资料
- https://www.postgresql.org/docs/current/
- 内部 RFC：电子商务平台的数据库选择
```

## 快速参考

| 章节 | 目的 | 关键问题 |
|---------|---------|--------------|
| 状态 | 当前状态 | 这个决策当前有效吗？ |
| 背景 | 背景信息 | 我们为什么要做出这个决定？ |
| 决策 | 选择结果 | 我们选择了什么？ |
| 后果 | 影响 | 现在会发生什么？ |
| 备选方案 | 其他选项 | 还考虑了哪些其他方案？ |
